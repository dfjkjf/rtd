## 任务调度器

通过单线程来高效管理多个周期性任务，而不必为每个任务单独创建线程。这种模式通常被称为 **"任务调度器"** 或 **"时间轮"**，是高性能编程中常见的设计模式。以下是关键原理和实现思路：

---

### 1. **核心思想**
- **统一调度**：所有周期性任务被注册到一个**全局队列**，由单个线程（常称 `EventLoop`）统一检查并触发。
- **时间排序**：任务按下次执行时间排序（如用小根堆或时间轮），每次只处理当前时间到期的任务。
- **非阻塞睡眠**：线程在无任务时休眠，直到下一个任务的触发时间（避免忙等待）。

---

### 2. **典型实现方式**
#### 方法一：优先级队列（小根堆）
```python
import heapq
import time

class Scheduler:
    def __init__(self):
        self.tasks = []
    
    def add_task(self, func, interval):
        # 下次执行时间 = 当前时间 + 间隔
        heapq.heappush(self.tasks, (time.time() + interval, func, interval))
    
    def run(self):
        while self.tasks:
            next_time, func, interval = self.tasks[0]
            now = time.time()
            if now >= next_time:
                heapq.heappop(self.tasks)
                func()  # 执行任务
                self.add_task(func, interval)  # 重新加入队列
            else:
                time.sleep(max(0, next_time - now - 0.001))  # 避免过度休眠
```

#### 方法二：时间轮（Timing Wheel）
- 更适用于高频任务（如延迟消息队列），通过环形数组分桶管理任务。

---

### 3. **为什么比多线程更优？**

| 维度       | 单线程调度器               | 多线程方案               |
|----------------|-----------------------------|----------------------------|
| **资源占用**    | 1个线程，低内存/CPU开销        | n个线程，上下文切换成本高      |
| **精度控制**    | 统一时间管理，误差可控          | 线程调度受系统影响，误差累积    |
| **线程安全**    | 无需锁（天然单线程）            | 需处理同步问题               |
| **扩展性**      | 动态增删任务容易               | 线程生命周期管理复杂           |

---

### 4. **适用场景**
- **高频小任务**：如定时心跳检测、数据采样。
- **延迟任务**：如游戏技能冷却、订单超时处理。
- **IO密集型**：配合异步IO（如asyncio）实现高并发。

---

### 5. **注意事项**
- **长任务阻塞**：若某个任务执行时间过长，会影响其他任务触发时机。解决方案：
    - 将耗时任务交给线程池。
    - 使用 `asyncio` 等异步框架。

    | 原则 | 说明 |
    |------|------|
    | **不要让调度器等待** | 调度器主循环必须是非阻塞的 |
    | **区分任务类型** | I/O密集型 用异步，CPU密集型 用多进程，阻塞用线程池 |
    | **设置超时** | 每个任务都应该有超时保护 |
    | **监控队列长度** | 线程池/事件循环队列堆积时需要告警 |

- **时间漂移**：任务的执行间隔是“固定延迟”而非“固定频率”（实际间隔=执行时间+设定间隔）。若需严格频率，需动态调整下次触发时间。
    - 问题复现：do_work() + sleep(period) ❌
    - 固定起点 + 动态计算 sleep 时间： do_work() + sleep(next_time-current_time) ✅

## 优先级反转（Priority Inversion）


优先级反转最可怕的地方在于，它会导致**高优先级任务无故卡死**，且这种 Bug 往往具有随机性，极难调试。
* **低优先级**任务占着资源。
* **中优先级**任务抢占了 CPU，导致低优先级任务无法释放资源。
* **高优先级**任务被迫无限等待。

## 3. 区分“理论派”与“工程派”
解决优先级反转通常有两种标准手段，面试官通常会看你能不能准确描述它们：
* **优先级继承协议（Priority Inheritance）：** 让占有资源的任务临时提升到高优先级。
* **优先级天花板协议（Priority Ceiling）：** 预先给资源设定一个“最高优先级”。

在操作系统（尤其是实时操作系统 RTOS，如 FreeRTOS、RT-Thread、VxWorks）中，**优先级继承**和**优先级天花板**并不是简单的“函数接口”，而是**互斥锁（Mutex）底层的一种调度策略或配置属性**。

通常你通过调用创建互斥锁的 API，并传入特定的参数来启用它们。

---

### 1. 优先级继承协议 (Priority Inheritance Protocol)

#### 原理
当高优先级任务（High）请求一个被低优先级任务（Low）持有的锁时，内核会临时将 **Low 的优先级提升到与 High 一致**。
* **目的**：防止中等优先级任务（Mid）抢占 Low 的 CPU 时间，从而让 Low 尽快运行完并释放锁。
* **恢复**：一旦 Low 释放了锁，它的优先级会立即恢复到原始水平。



#### 怎么使用？
在大多数 OS 中，这是**互斥锁（Mutex）的默认行为**。
* **POSIX (Linux/Unix):** 使用 `pthread_mutexattr_setprotocol` 设置为 `PTHREAD_PRIO_INHERIT`。
* **FreeRTOS:** 它的 `Mutex` 默认就实现了优先级继承，你只需要使用 `xSemaphoreCreateMutex()` 即可。
* **代码逻辑：** 开发者正常 `Lock` 和 `Unlock`，提升优先级的过程由 OS 内核在后台自动完成。

---

### 2. 优先级天花板协议 (Priority Ceiling Protocol)

#### 原理
给每个资源（锁）设定一个固定的“天花板优先级”。这个优先级通常设置为**可能访问该资源的所有任务中最高的那个**。
* **机制**：只要任务获取了这个锁，不管它原本优先级多低，都会**立即提升**到天花板优先级。
* **优势**：它比优先级继承更激进，不仅能解决优先级反转，还能在一定程度上**预测任务执行时间**，并减少上下文切换次数。


#### 怎么使用？
这通常需要开发者在创建锁时手动指定一个优先级数值。
* **POSIX:** 设置属性为 `PTHREAD_PRIO_PROTECT`，并用 `pthread_mutexattr_setprioceiling` 指定高度。
* **RTOS 示例：** 1. 确定访问资源 A 的有 Task1(prio 5) 和 Task2(prio 10)。
  2. 创建互斥锁时，设置其 Ceiling 为 10。
  3. 当 Task1 获取锁的一瞬间，它立刻变成 prio 10，直到放锁。

---

### 3. 核心区别对比

| 特性 | 优先级继承 (Inheritance) | 优先级天花板 (Ceiling) |
| :--- | :--- | :--- |
| **触发时机** | **被动触发**：只有高优先级任务来抢锁时才提速。 | **主动触发**：只要拿到锁，立刻提速。 |
| **预判要求** | 无需预知，OS 动态处理。 | **需要预知**：开发者必须知道哪些任务会用这个锁。 |
| **开销** | 涉及多次上下文切换，逻辑较复杂。 | 逻辑简单，但可能导致不必要的“高优先级”运行。 |
| **防止死锁** | 只能缓解优先级反转。 | 如果设计得当，可以完全避免死锁。 |

---

### 总结建议
* **如果你追求简单和通用**：直接用 **Mutex**（大多数 OS 默认支持优先级继承）。
* **如果你在做极其严苛的实时控制（如航天、车载控制）**：推荐使用 **优先级天花板**，因为它更稳定，能提供更好的确定性（Deterministic），避免了由于“等待抢占”带来的抖动。

优先级继承协议，无法解决死锁问题，必须固定顺序获取锁
天花板协议规定：一个任务只有在它的优先级严格高于当前所有被占用资源的天花板时，才能获取新资源。

## 调度

你说得非常形象！如果把操作系统比作一个**大管家**，那么“调度”（Scheduling）就是这个管家的**排班表**。

简单来说，调度解决的就是：**在资源（主要是 CPU）有限的情况下，按照什么样的策略，决定哪个任务（进程/线程）在什么时候占用 CPU 运行多久。**

---

### 1. 调度的“三大要素”

就像你说的，“什么时间安排什么人干什么活”，操作系统调度也离不开这三个维度：

#### **“什么人” (Subject)**
在操作系统眼里，人就是**进程（Process）**或**线程（Thread）**。
* 有些任务是“计算密集型”（一直在算数，费 CPU）。
* 有些任务是“I/O 密集型”（一直在等用户点鼠标或等网络下载，不费 CPU 但费时间）。

#### **“什么时间” (Timing)**
这涉及到调度的**触发点**。管家不会无缘无故换人，通常在以下时刻换班：
* **任务主动放弃**：干完活了，或者要去休息（等待数据）。
* **时间片到了**：每个任务只准干 10 毫秒，到点强制下场（分时系统）。
* **插队（抢占）**：来了一个更紧急的任务（比如系统告警），必须马上处理。

#### **“干什么活” (Execution)**
就是执行代码逻辑。调度器不关心代码的具体内容，它只关心这活儿的**优先级**和**状态**。

---

### 2. 调度是怎么“安排”的？（经典算法）

管家手里有一套“规章制度”，这就是调度算法。常见的有几种：

| 调度算法 | 形象比喻 | 特点 |
| :--- | :--- | :--- |
| **先来先服务 (FCFS)** | 银行排队，谁先来谁先办。 | 公平，但如果排头的是个“慢郎中”，后面人都得饿死。 |
| **短作业优先 (SJF)** | 谁的活儿快谁先上。 | 效率最高，但长任务可能会永远排不上号（饥饿现象）。 |
| **时间片轮转 (RR)** | 每个人只给 3 分钟，没干完就去队尾重新排。 | 响应快，适合多用户交互。 |
| **优先级调度** | VIP 优先，普通客户靠后。 | 灵活，但容易产生你之前问的“优先级反转”问题。 |
| **多级反馈队列** | 综合以上所有。干得快的去高频队，干得慢的降级。 | **现代操作系统（如 Linux、Windows）的主流做法。** |

---

### 3. 为什么需要调度？

如果没有调度，计算机会变成什么样？
1.  **卡死**：一个程序写了个死循环，你的电脑就彻底瘫痪了，因为别的程序抢不到 CPU。
2.  **浪费**：CPU 算力极快，如果没有调度，当程序在等硬盘读数据时，CPU 就会在那“发呆”，白白浪费电。
3.  **没法多任务**：你不能一边听歌一边写代码，因为 CPU 同时只能干一件事。

---

### 4. 调度的本质：一种权衡

操作系统的调度器其实是个**端水大师**。它要在以下几个目标之间找平衡：
* **吞吐量**：单位时间内干的活越多越好。
* **周转时间**：从任务开始到结束的时间越短越好。
* **响应时间**：我点一下鼠标，你得立刻理我（实时性）。

**一句话总结：**
调度就是操作系统在**公平**（每个人都有份）和**效率**（不让 CPU 闲着）之间，通过一套复杂的数学算法进行的“资源分配艺术”。
