# 并发编程的常见问题

## 1. 数据竞争与未定义行为

### 问题表现
- 多个线程同时读写同一变量导致未定义行为
- 看似正常但偶尔崩溃或结果错误

### 解决方案
```cpp
// 错误示例
int counter = 0;
std::thread t1([&]{ for(int i=0;i<10000;i++) counter++; });
std::thread t2([&]{ for(int i=0;i<10000;i++) counter++; });

// 正确示例：使用原子操作
std::atomic<int> counter{0};

// 或使用互斥锁
std::mutex mtx;
int counter = 0;
mtx.lock(); counter++; mtx.unlock();
```

## 2. 伪共享

### 问题表现
- 性能远低于预期
- CPU缓存行失效频繁

### 解决方案
```cpp
// 错误：相邻的原子变量共享缓存行
struct BadStruct {
    std::atomic<long> a;
    std::atomic<long> b;  // 与a在同一缓存行(64字节)
};

// 正确：缓存对齐
struct alignas(64) GoodStruct {
    std::atomic<long> a;
    char padding[56];  // 填充到64字节
    std::atomic<long> b;
};
```

## 3. 锁竞争与死锁

### 问题表现
- 线程大量时间在等待锁
- 程序偶尔完全卡死

### 解决方案
```cpp
// 死锁示例
std::mutex m1, m2;
void thread1() { 
    std::lock_guard l1(m1); 
    std::lock_guard l2(m2); 
}
void thread2() { 
    std::lock_guard l2(m2); 
    std::lock_guard l1(m1); 
}

// 解决方案1：固定顺序获取锁
void thread1() { 
    std::lock(m1, m2);
    std::lock_guard l1(m1, std::adopt_lock);
    std::lock_guard l2(m2, std::adopt_lock);
}

// 解决方案2：使用std::scoped_lock (C++17)
void safe_thread() {
    std::scoped_lock lock(m1, m2);  // 一次性原子地获取多个锁，从底层避免死锁
}
```

## 4. 任务窃取与负载不均

### 问题表现
- 部分核心空闲，部分核心过载
- 整体吞吐量低

### 解决方案
使用工作窃取队列：
```cpp
// 使用Intel TBB或类似库
#include <tbb/parallel_for.h>
tbb::parallel_for(0, 1000000, [](int i) {
    // 自动负载均衡的工作
});

// 或实现工作窃取队列
class WorkStealingQueue {
    // 每个线程有自己的deque
    // 空闲线程从忙碌线程的deque尾部窃取任务
};
```

## 5. 内存顺序错误

### 问题表现
- 使用原子操作但仍有奇怪的可见性问题
- 指令重排导致逻辑错误

### 解决方案
```cpp
// 错误：过于宽松的内存顺序
std::atomic<bool> ready{false};
int data = 0;
// 线程1
data = 42;
ready.store(true, std::memory_order_relaxed);  // 可能被重排到data赋值之前！

// 正确：使用合适的内存顺序
// 生产者
data = 42;
ready.store(true, std::memory_order_release);  // 保证之前的写入可见

// 消费者
while(!ready.load(std::memory_order_acquire));  // 保证之后的读取可见
assert(data == 42);  // 安全
```

## 6. 线程创建开销过大

### 问题表现
- 频繁创建销毁线程导致性能下降
- 响应延迟高

### 解决方案
使用线程池：
```cpp
class ThreadPool {
    std::vector<std::thread> workers;
    std::queue<std::function<void()>> tasks;
    
public:
    template<typename F>
    auto enqueue(F&& f) -> std::future<decltype(f())> {
        // 提交任务到队列，由工作线程执行
    }
};

// 使用示例
ThreadPool pool(8);  // 8个工作线程
auto future = pool.enqueue([]() { return compute(); });
```

## 7. 优先级反转

### 问题表现
- 高优先级线程被低优先级线程阻塞
- 系统响应性下降

### 解决方案
```cpp
// 使用优先级继承的互斥锁
std::mutex mtx;  // 普通mutex不支持优先级继承

// 在某些平台上使用PTHREAD_PRIO_INHERIT
pthread_mutexattr_t attr;
pthread_mutexattr_setprotocol(&attr, PTHREAD_PRIO_INHERIT);

// 或使用更高级的同步原语，避免长时间持锁
```

## 8. False Sharing的检测与优化

### 诊断工具
```bash
# Linux perf检测缓存未命中
perf stat -e cache-misses ./your_program

# Intel VTune分析缓存行争用
# 使用pcm (Performance Counter Monitor)工具
```

### 优化模式
```cpp
// 读多写少场景：使用共享锁
std::shared_mutex rw_mutex;

// 写时复制(Copy-on-Write)
class COWData {
    std::shared_ptr<const Data> ptr;
    std::mutex mtx;
    
    void modify() {
        std::lock_guard lock(mtx);
        if (ptr.use_count() > 1) {
            ptr = std::make_shared<Data>(*ptr);  // 复制
        }
        // 修改独占副本
    }
};
```

## 9. ABA问题

### 问题表现
- 使用CAS(Compare-And-Swap)时，值从A→B→A，误认为未改变

### 解决方案
```cpp
// 使用带标签的指针
struct TaggedPtr {
    void* ptr;
    uint64_t tag;  // 每次修改时递增
};

std::atomic<TaggedPtr> atomic_ptr;

// 或使用std::atomic<std::shared_ptr> (C++20)
std::atomic<std::shared_ptr<MyClass>> atomic_sp;
```

## 10. 性能监控关键指标

| 指标 | 含义 | 优化目标 |
|------|------|----------|
| 锁争用率 | 线程等待锁的时间比例 | < 5% |
| 缓存未命中率 | L3缓存未命中比例 | < 10% |
| 上下文切换速率 | 每秒上下文切换次数 | < 10000 |
| 指令退役率 | IPC(每周期指令数) | > 1.0 |

## 最佳实践总结

1. **尽量使用高层抽象**：`std::async`、`std::parallel_algorithms`、TBB
2. **最小化共享**：优先使用线程局部存储
3. **无锁设计优先**：消息传递(actor模型)优于共享内存
4. **性能测试驱动**：使用Google Benchmark、perf等工具
5. **编译器屏障**：`std::atomic_signal_fence`、`asm volatile("" ::: "memory")`
